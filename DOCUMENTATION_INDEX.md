## Documentation Update Guidelines

### When to Update

1. **README.md**
   - New major features added
   - Installation process changes
   - Project description updates
   - Version milestones

2. **CHANGELOG.md**
   - **EVERY release** (mandatory)
   - Bug fixes
   - New features
   - Breaking changes

3. **TESTING_SUMMARY.md**
   - New test modules completed
   - Major testing milestones
   - Coverage improvements
   - New testing infrastructure

4. **UNIT_TESTING.md**
   - New testing patterns introduced
   - Testing tools added/changed
   - Best practices updates

5. **ARCHITECTURE.md**
   - Architectural changes
   - New design patterns
   - Dependency updates

### Update Checklist

Before releasing a new version:

- [ ] Update `CHANGELOG.md` with version details
- [ ] Update `TESTING_SUMMARY.md` if tests added
- [ ] Update `README.md` if features changed
- [ ] Verify all documentation links work
- [ ] Check for broken cross-references
- [ ] Update "Last Updated" dates
- [ ] Review this index for new docs

## Avoiding Documentation Duplication

### Single Source of Truth

- **Testing**: UNIT_TESTING.md (detailed) + TESTING_SUMMARY.md (overview)
- **Architecture**: ARCHITECTURE.md only
- **Contributing**: CONTRIBUTING.md only
- **Features**: README.md (list) + CHANGELOG.md (history)

### Cross-Reference Guidelines

When information exists in multiple docs:
- ‚úÖ **Good**: Link to primary source
- ‚ùå **Bad**: Copy-paste same content

Example:
```markdown
<!-- In README.md -->
For detailed testing information, see [UNIT_TESTING.md](./UNIT_TESTING.md).

<!-- NOT this: -->
<!-- Copying entire testing guide into README -->
```
